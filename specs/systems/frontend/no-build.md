# No-build frontend stack

This stack builds server-rendered web apps with no bundler, transpiler, or npm build step. The server is the source of truth for all state; the frontend is HTML with behavioral attributes.

## How the pieces fit

| Tool | Role |
|---|---|
| **htmx** | Server interactions — replaces `fetch`/XHR with HTML attributes |
| **_hyperscript** | Client-side behavior — replaces JavaScript for local interactivity |
| **Bulma** | Styling — CSS-only, no JavaScript component library needed |
| **dcdn** | Vendor asset management — pulls files from npm packages locally, no CDN links in templates |

## Constraints

**Server responses are HTML, not JSON.** htmx consumes HTML fragments returned by the server. Do not build JSON API endpoints for frontend consumption; the server renders and returns markup directly.

**No JavaScript files or script tags for application logic.** htmx handles anything that needs the server; _hyperscript handles anything that is purely client-side. If you reach for a `<script>` tag, ask whether htmx or _hyperscript already covers the need.

**No frontend framework alongside this stack.** Do not introduce React, Vue, Alpine, or similar. They are incompatible with the no-build constraint and redundant given htmx + _hyperscript.

**Vendor assets are local, not CDN-linked.** Use `dcdn add` to pull files from npm packages into the output directory. Do not reference unpkg, jsdelivr, cdnjs, or similar in templates. Commit `dcdn.json`, `package.json`, and `bun.lock`; do not commit `node_modules/` or the output directory.

## Enforcement rules

**htmx**
- Use htmx for all server interactions; do not use `fetch`, `XMLHttpRequest`, or any JS HTTP client
- JavaScript that performs its own network requests is only acceptable for third-party integrations (e.g. analytics, payment SDKs) with no htmx equivalent

**_hyperscript**
- Use _hyperscript for all client-side behavior; do not write standalone JavaScript
- The only acceptable JS is code embedded inside a `js` block within a _hyperscript handler
- Do not add `<script>` tags or `.js` files for client-side logic
- Do not use `addEventListener`, `querySelector`, or other vanilla JS DOM APIs outside of a _hyperscript `js` block

**Bulma**
- Use Bulma for all styling; do not introduce a second CSS framework (Tailwind, Bootstrap, Foundation, etc.)
- Reach for Bulma modifier classes before writing custom CSS; only write custom CSS when Bulma has no equivalent

**dcdn**
- Use dcdn for all frontend vendor assets; do not reference CDN URLs (unpkg, jsdelivr, cdnjs, etc.) directly in templates
- Commit `dcdn.json`, `package.json`, and `bun.lock`; do not commit `node_modules/` or the output directory — both are regenerated by `dcdn install`
- Run `dcdn install` after cloning or pulling to restore vendor assets before running the project
